# 多候选品横向与组合决策

## 目录

- 核心原则

- 确定性脚本
- 输出协议

## 核心原则

先使用完全相同的证据截止日、国家/平台口径、五道门槛和七维模型分别评估每个候选，再进行组合选择。组合层不得修改任何候选的基础总分；它只处理置信度、预算、运营能力、集中度、协同和风险暴露。

若候选不在同一国家、平台或成熟阶段，先分组或标准化口径，禁止把“已有销量老品”和“概念阶段新品”的分数直接排名。

## 三层决策

### 1. 硬淘汰

先排除：合规/IP/安全红线、压力场景无修复路径的亏损、不可履约、供应商关键资料缺失、超过绝对预算或与卖家能力明显不匹配的候选。硬淘汰品不参与组合优化。

### 2. 单品可比排名

| 字段 | 用途 |
|---|---|
| 基础总分与七维分 | 保留核心模型结论 |
| 置信度 | 降低弱证据高分的优先级 |
| 初始投资 | 样品、合规、首单、包装、头程与启动营销 |
| 运营负荷 | 内容、客服、变体、质检、认证和库存复杂度 |
| 现金周期 | 预付款到回款的预计时间 |
| 关键资源 | 平台账号、达人、供应商、认证或团队能力 |

除总分外，至少报告“单位预算价值”和最大短板。不要使用总分除以投资作为唯一排名，因为低成本弱机会可能被错误抬高。

### 高分平局裁决

当多个候选均达到 80+ 分且整体置信度为高时，不再用 1-2 分的总分差异直接决定优先级。先确认各候选没有门槛红线，再按以下顺序打破平局：

1. 现金周转速度：优先选择预付款少、交期短、库存周转快、回款路径清晰的候选。
2. 运营复杂度：优先选择内容、客服、变体、质检、认证和售后负荷更低的候选。
3. 风险相关性：优先选择与现有组合在平台、供应商、季节、流量来源和合规风险上相关性更低的候选。
4. 供应链确定性：优先选择报价、样品、包装尺寸、质检点和替代供应商更清晰的候选。
5. 学习价值：优先选择能验证新平台、新人群、新价格带或新内容假设的候选。

若平局仍无法打破，输出并列优先级，并设计同预算、同周期的小样本并行测试。不要为了制造单一冠军而篡改基础分。

### 3. 组合优化

组合应满足：SKU 数量、总预算、运营负荷、品类/平台集中上限和红线约束。再比较：

- 收益质量：基础分、利润与置信度；
- 风险相关性：是否同时依赖同一平台、供应商、季节或流量来源；
- 资源协同：供应链、内容素材、目标人群、认证和库存能否复用；
- 现金节奏：避免所有候选同时长周期占款；
- 学习价值：首轮组合能否验证不同假设，而非三个高度相似产品。

### 多目标辅助排序

当候选较多、且每个候选都已有基础 CIDM 评分时，可以增加多目标辅助排序，但它只用于解释资源分配和排序微调，不得覆盖基础总分、门槛红线和人工复核结论。

辅助目标可按场景选择：

| 目标 | 示例指标 | 使用限制 |
|---|---|---|
| 转化潜力 | CTR、CVR、加购率、内容点击率、历史素材表现 | 只和同平台同阶段候选比较 |
| 利润质量 | 单件贡献利润、盈亏平衡 ACOS/CPA、批次回本订单量 | 必须使用同币种和同成本口径 |
| 物流/履约 | 尺寸重量、运费敏感性、签收时效、仓储周转 | 不得用低物流成本掩盖需求弱 |
| 风险暴露 | 合规/IP、海关、退货、账号处罚、供应商集中度 | 红线风险先硬淘汰，再排序 |
| 学习价值 | 新人群、新平台、新价格带、新内容假设 | 只能作为小预算测试理由 |
| 反馈分布 | 多达人/多素材/多地区是否都有效 | 样本不足时只作观察信号 |

建议输出“基础分 + 辅助排序理由”，而不是生成新的黑箱综合分。例如：

```text
候选 A 基础分最高，但物流成本和退货敏感；候选 B 基础分略低，但现金周期短、达人端口分布更健康。因此组合中保留 A 做单品冠军，同时让 B 获得小预算并行测试位。
```

如果确实需要加权辅助效用，必须披露权重，并写明“该效用不是 CIDM 总分”。权重应来自本轮资源约束，例如现金紧张时提高现金周期权重，内容团队强时提高内容复用权重，合规不确定时提高风险暴露权重。

### 4. 组合级盈亏平衡

组合选择前还要回答一个前置问题：测品系统本身是否赚钱。单品分数高不代表月度测品池可持续；如果每测一个品的成本过高，或成功品生命周期收益过低，即使选中概率看似不错也可能亏损。

核心公式：

`最低成功率 = 单测品成本 ÷ 单成功产品预期总收益`

`预期组合利润 = 测品数量 × 预期成功率 × 单成功产品预期总收益 - 测品数量 × 单测品成本`

单测品成本至少包括：样品、采购、物流、素材、达人/投流、人力、管理和必要工具费用。单成功产品预期总收益必须写明月利润、生命周期、收益达成度和主要假设，不得直接填乐观 GMV。

判断方式：

- 历史成功率高于最低成功率，且样本口径相似：可以继续按组合优化选品。
- 历史成功率接近最低成功率：收紧候选门槛，降低测品成本或缩短测试周期。
- 历史成功率低于最低成功率：不要扩大测品数量；先提高切入楔子质量、降低固定成本或改平台/渠道。
- 成功产品收益高度不确定：做 `-30% / 基准 / +30%` 敏感性，观察最低成功率是否跨过团队历史能力线。

结构化输入时运行：

`python3 scripts/portfolio_break_even.py input.json`

## 组合健康度诊断

在对现有产品组合做新品补充之前，先对组合本身做健康度体检。不健康的组合不应该盲目加新品——先修复结构问题，再考虑补充。

### 诊断维度

| 维度 | 健康信号 | 警告信号 | 危险信号 |
|---|---|---|---|
| 集中度 | 单一品类/平台/供应商占比 <50% | 单一品类/平台占比 50-70% | 单一品类/平台/供应商占比 >70% |
| 现金流节奏 | 不同 SKU 的回款周期错开，现金流平稳 | 多数 SKU 同时占款，偶有现金流紧张 | 持续需要外部资金或其他 SKU 利润维持 |
| 风险相关性 | SKU 之间风险因子分散（不同平台/供应商/季节/流量来源） | 多数 SKU 共享同一风险因子（如同一供应商或同一平台） | 所有 SKU 高度相关，一损俱损 |
| 生命周期分布 | 成长品 30-40%、成熟品 40-50%、新品持续引入 | 成熟品 >70%（增长乏力）或成长品 <20%（缺乏增长动力） | 衰退品 >30%（需要清仓）且无新品管线 |
| 利润结构 | 利润品占 30-50%，引流品占 20-30%，亏损品 <10% | 利润品 <25% 或亏损品 10-25% | 亏损品 >25% 或利润品 <15% |
| 运营负荷 | 团队负荷在 70% 以下，有余力处理异常 | 团队负荷 70-85%，偶有加班 | 团队负荷 >85%，频繁出错或遗漏 |

### 诊断流程

1. **盘点现有组合**：列出所有 SKU 的品类、平台、供应商、生命周期阶段、利润贡献、运营负荷。
2. **逐维度评估**：按上述 6 个维度打分（绿/黄/红）。
3. **识别结构问题**：如果有红色维度，先修复再考虑加新品。
4. **确定补充方向**：基于诊断结果，明确新品应该补充什么（分散风险？填充价格带？引入增长品？替换衰退品？）。
5. **设定补充约束**：基于当前运营负荷和现金流，确定新品补充的数量和预算上限。

### 诊断输出

组合健康度诊断的输出包括：

- 6 维度的体检结果（绿/黄/红 + 具体数据）。
- 结构问题清单（哪些维度需要优先修复）。
- 新品补充方向（应该补什么类型的品，不应该补什么类型）。
- 补充约束（数量上限、预算上限、运营负荷上限）。

这个输出直接作为组合选择（三层决策）的输入约束。

## 确定性脚本

需要在明确预算和数量约束下选择组合时运行：

`python3 scripts/portfolio_selector.py input.json`

输入示例：

```json
{
  "constraints": {
    "desired_items": 3,
    "budget": 50000,
    "max_operational_load": 10,
    "max_per_category": 2,
    "max_per_platform": 2
  },
  "candidates": [
    {
      "id": "A",
      "name": "候选 A",
      "score": 78,
      "confidence": "high",
      "dimension_confidence": {
        "market_demand": "确定",
        "entry": "中等",
        "profit": "确定",
        "content": "中等",
        "supply": "确定",
        "risk": "中等",
        "window": "中等"
      },
      "investment": 12000,
      "operational_load": 3,
      "cash_cycle_days": 45,
      "category": "beauty",
      "platform": "amazon",
      "supplier": "S1",
      "redline": false
    }
  ]
}
```

`score` 必须来自原七维模型。`confidence` 只允许 `high/medium/low`。`dimension_confidence` 用于人工解释和复核，脚本当前不强制读取。运营负荷使用同一团队定义的相对尺度。脚本输出的是“组合选择效用”，不是新的投资总分。

## 输出协议

1. 评估边界与资源约束。
2. 硬淘汰清单及原因。
3. 单品横向表：基础分、整体置信度、维度置信度短板、投资、负荷、现金周期、核心短板。
4. 推荐组合：入选候选、总投资、总负荷、组合效用和集中度。
5. 入选原因：每个候选贡献什么、与其他候选如何协同或分散风险。
6. 落选原因与候补顺序：区分“产品差”与“当前资源不匹配”。
7. 组合压力测试：预算缩减、广告恶化、一个供应商失效或销售窗口缩短。
8. 分阶段资金释放：不要一次性给所有入选品全额预算，按验证里程碑解锁。

最终表述同时给出“单品冠军”和“组合入选”。最高分产品不一定属于最优组合，但任何组合调整都不能掩盖其基础评分和红线。
