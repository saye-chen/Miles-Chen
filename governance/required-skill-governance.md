# 八域 Skill 治理文件标准

## 目的

本标准把“文件存在”“合同可运行”“专业知识充分”“真实经营验证”拆开验收。行数只能发现异常，不能证明专业性；任何 `100/100` 必须写明所属层级，禁止把结构分冒充专家成熟度或生产效果。

## 四个必需治理模块

每个专业域必须直接提供以下语义模块。允许根目录同名文件兼容，但标准路径必须存在并在 `SKILL.md` 直接路由：

1. `references/professional-depth-governance.md`：专业不变量、升级条件、证据降级、反事实、停止和结果回收。
2. `references/skill-integration-protocol.md`：输入/输出卡、双向主权、版本、部分失败、冲突裁决和回写状态机。
3. `references/output-protocols/professional-report-delivery.md`：Decision Card、Decision Memo、Diligence 的最低内容、压缩边界与验收。
4. `references/data-contract-and-automation.md`：对象、字段、缺失/零值、时区币种、幂等、血缘、权限、失败与外部写入。

## 语义最低门

每个治理文件必须包含并实质定义以下主题，而非只出现关键词：

- 对象、版本、`as_of_time` 和适用边界；
- observed/authorized/derived/inferred/causal 的证据等级；
- 主权 owner、参与域、允许用途、禁止用途；
- 输入、输出、缺失、冲突、过期和部分失败；
- 动作、责任人、成功、停止、回滚、退出和结果回填；
- 机器校验入口或明确的人工复核条件。

专业 reference 不设统一 50 行硬门。核心模块少于 50 行时必须通过语义密度检查：至少覆盖对象、机制、计算或判定、约束、证据、反事实、动作、成功/停止/回滚/退出九类中的七类；平台卡还必须有竞价/交付、优化反馈、归因、平台特有失败模式和动态核验清单。

## 四层成熟度

| 层级 | 可声明内容 | 必需证据 |
|---|---|---|
| L1 Structure | 目录、frontmatter、死链、基础入口通过 | quick validation |
| L2 Contract | 对象、主权、证据、计算和失败状态可机器验证 | 域级与共享合同测试 |
| L3 Expert | 核心知识、平台差异、反事实和域级模型通过独立压力测试 | 完整 Golden、对抗、性质测试 |
| L4 Production | 对真实经营对象有效且可回放 | 至少3个授权、脱敏、含结果和哈希的历史案例 |

仓库总分不得跨层平均。L1/L2 全通过但 L3 未通过时只能称 `contract ready`；L4 未通过时只能称 `controlled pilot`。

## 变更与版本

- 勘误且不改变动作不升级运行时。
- 新增知识、场景或非破坏性脚本时升级发布年月并记录影响闭包。
- 输入结构、评分、决策档位、主权或输出合同发生破坏性变化时提升 major、保留旧版本重算器并禁止跨版本直接比较。
- 版本号不得作为成熟度证据；发布说明必须同时列出 L1—L4 状态。

## 发布门

发布前运行统一治理验证、每域 quick validation、域级测试、共享跨域测试、对抗测试和完整报告评分。任何缺失治理模块、伪造数据、越权动作、版本污染、幻觉数字、非有限数、守恒失败或真实回放冒充均为阻断项。
